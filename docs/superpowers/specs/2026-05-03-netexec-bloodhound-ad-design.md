# Design: NetExec + BloodHound + Per-Target KB (AD Capability Pack)

**Date:** 2026-05-03
**Status:** Approved (brainstorming complete; ready for implementation plan)
**Roadmap items:** Top 5 #1 (NetExec/BloodHound/cypher) + Top 5 #4 (per-target KB) — bundled
**Out of scope:** Top 5 #2 (searchsploit/msfvenom), #3 (Playwright wiring), #5 (hypothesis-loop prompt restructure)

## Goal

Add elite-tier Active Directory engagement capability to reverser by wrapping
NetExec (per-protocol), BloodHound (Neo4j lifecycle + bloodhound-python collector
+ canned cypher queries + free-form cypher), and a persistent per-target
knowledge base (SQLite) that all AD-relevant tools read from and write to.

The KB foundation also serves item #4 of the capability roadmap, pulled forward
to avoid building credentials handling twice.

## Background

The current pentest profile reaches a wall on internal AD engagements: there is
no NetExec, no BloodHound, no graph-based attack-path planning, and no
persistent state across tool calls — the LLM either burns tokens re-passing
credentials or forgets them. The 10.13.38.23 engagement captured in
`pentest_report_10.13.38.23.md` shows the failure mode: ~1700 password attempts,
multiple CVE attempts, no foothold, no lessons retained between rounds.

This work closes the largest single capability gap identified in
`CAPABILITY_ROADMAP.md` and lays the persistent-state foundation that several
other roadmap items depend on.

### Per-target on-disk layout

```
targets/<target>/
├── state.db          # SQLite KB
├── neo4j/            # Neo4j data dir for this target's BloodHound graph
├── findings/         # evidence files referenced by `findings.evidence_paths`
├── loot/             # dumped hashes, tickets, NTDS, files extracted via spider
├── scope.toml        # OPTIONAL: in-scope CIDRs, time windows, no-DoS flags
└── report.md         # generated by kb_export_report
```

`<target>` is the normalized (lowercased, trimmed) target identifier passed by
the LLM as the `target` argument — IP, hostname, or CIDR. The `targets/`
parent directory lives at the project root (or `REVERSER_TARGETS_DIR` if set).

### KB schema

```sql
CREATE TABLE targets (
    id           TEXT PRIMARY KEY,    -- normalized target identifier
    hostname     TEXT,
    ip           TEXT,
    domain       TEXT,
    scope_notes  TEXT,
    first_seen   TEXT NOT NULL,        -- ISO 8601
    last_active  TEXT NOT NULL
);

CREATE TABLE hosts (
    target_id     TEXT NOT NULL REFERENCES targets(id),
    ip            TEXT NOT NULL,
    hostname      TEXT,
    os            TEXT,
    domain        TEXT,
    is_dc         INTEGER DEFAULT 0,    -- boolean
    smb_signing   TEXT,                 -- "required" | "enabled" | "disabled" | NULL
    first_seen    TEXT NOT NULL,
    PRIMARY KEY (target_id, ip)
);

CREATE TABLE services (
    target_id     TEXT NOT NULL REFERENCES targets(id),
    host_ip       TEXT NOT NULL,
    port          INTEGER NOT NULL,
    proto         TEXT NOT NULL,        -- "tcp" | "udp"
    service       TEXT,                 -- "ssh" | "http" | "smb" | ...
    version       TEXT,
    banner        TEXT,
    scan_source   TEXT,                 -- which tool reported this
    scanned_at    TEXT NOT NULL,
    PRIMARY KEY (target_id, host_ip, port, proto)
);

CREATE TABLE credentials (
    id                INTEGER PRIMARY KEY AUTOINCREMENT,
    target_id         TEXT NOT NULL REFERENCES targets(id),
    username          TEXT NOT NULL,
    password          TEXT,
    nt_hash           TEXT,
    lm_hash           TEXT,
    kerberos_ticket   TEXT,             -- AS-REP / TGS hash blob
    domain            TEXT,
    source_tool       TEXT,             -- which tool reported this cred
    source_context    TEXT,             -- free-text (e.g., "asreproast against DC01")
    status            TEXT NOT NULL,    -- "untested" | "invalid" | "valid"
    first_seen        TEXT NOT NULL,
    last_tested       TEXT,
    UNIQUE (target_id, username, COALESCE(password, ''), COALESCE(nt_hash, ''))
);

CREATE TABLE cred_results (
    cred_id        INTEGER NOT NULL REFERENCES credentials(id),
    service_kind   TEXT NOT NULL,       -- "smb" | "winrm" | "ldap" | "mssql" | "ssh" | ...
    target_host    TEXT NOT NULL,
    success        INTEGER NOT NULL,    -- boolean
    error_msg      TEXT,
    attempted_at   TEXT NOT NULL,
    PRIMARY KEY (cred_id, service_kind, target_host, attempted_at)
);

CREATE TABLE findings (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    target_id       TEXT NOT NULL REFERENCES targets(id),
    title           TEXT NOT NULL,
    severity        TEXT NOT NULL,      -- "info" | "low" | "medium" | "high" | "critical"
    cvss            REAL,               -- optional numeric CVSS
    description     TEXT,
    evidence_paths  TEXT,               -- JSON array of paths under findings/ or loot/
    created_at      TEXT NOT NULL
);

CREATE TABLE artifacts (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    target_id     TEXT NOT NULL REFERENCES targets(id),
    kind          TEXT NOT NULL,        -- "asreproast_hashes" | "kerberoast_hashes" | "ntds_dump" | "discovered_paths" | ...
    path          TEXT NOT NULL,
    sha256        TEXT,
    source_tool   TEXT,
    created_at    TEXT NOT NULL
);

CREATE TABLE notes (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    target_id   TEXT NOT NULL REFERENCES targets(id),
    body        TEXT NOT NULL,
    created_at  TEXT NOT NULL
);
```

### Data flow

```
LLM tool call ──▶ tool wrapper ──▶ external binary (nxc, nmap, ...)
                       │
                       ├─▶ formatted display result ──▶ returned to LLM
                       │
                       └─▶ parser (pure fn) ──▶ kb.record_*() ──▶ SQLite
                                  │
                                  └─▶ failure ──▶ logged warning, tool unaffected

LLM tool call ──▶ kb_show / kb_list_* ──▶ SQLite read ──▶ formatted result
```

## New tool surface (LLM-facing)

### NetExec tools (`tools/netexec.py`) — 6 tools

All NetExec tools share a common arg base: `target` (IP/CIDR/file path),
`username`, `password`, `nt_hash`, `domain`, `local_auth` (bool),
`extra_args` (str). Per-protocol tools add an `action` enum.

```
netexec_smb(target, action, username?, password?, nt_hash?, domain?,
            local_auth=False, module?, ...)
  action: shares | users | groups | computers | pass_pol | rid_brute
        | sam | lsa | ntds | loggedon | sessions | disks | spider
        | exec | spray | check_auth
  module: optional NetExec module (lsassy, spider_plus, coerce_plus, enum_av, ...)

netexec_winrm(target, action, username?, password?, nt_hash?, domain?, command?)
  action: check_auth | exec | ps | spray

netexec_ldap(target, action, username?, password?, nt_hash?, domain?)
  action: check_auth | users | groups | computers | trusts | gmsa
        | asreproastable | kerberoastable | dc_list | active_users
        | admin_count | password_not_required

netexec_mssql(target, action, username?, password?, nt_hash?, domain?,
              query?, command?, local_auth=False)
  action: check_auth | databases | xp_cmdshell | query | spray

netexec_ssh(target, action, username?, password?, key_file?, command?)
  action: check_auth | exec | spray

netexec_ftp_wmi(target, protocol, action, username?, password?, command?)
  protocol: ftp | wmi
  action: check_auth | (ftp: list, get) | (wmi: exec)
```

**Cred fallback behavior:** If `username`/`password`/`nt_hash` are omitted, the
tool consults `credentials` table for valid creds for this target and uses the
most recently validated one. The chosen cred is logged in the tool's display
output (`[KB] Using credential: jdoe (validated 2m ago via netexec_smb)`).

**Spray safety guardrails (built into the tool, not just the prompt):**
- `netexec_*` `spray` actions hard-cap attempts per user at `REVERSER_SPRAY_MAX`
  env var (default: **3**). Override requires explicit env var change.
- `--continue-on-success` defaults to off.
- If `REVERSER_AD_ALLOW_SPRAY` is unset and the action is `spray`, the tool
  refuses with a clear error message explaining how to enable.

**KB writes:**
- Successful `check_auth` → `record_credential(status='valid')` + `record_cred_result(success=True)`
- Failed `check_auth` → `record_credential(status='invalid')` + `record_cred_result(success=False)`
- `users`/`groups`/`computers` (LDAP) → `record_host` for each computer entry
- `sam`/`lsa`/`ntds` dumps → `record_artifact(kind='ntds_dump' | 'sam_hashes' | 'lsa_secrets', path=auto-saved file under loot/)` + per-hash `record_credential(status='untested')`
- `shares` → `record_note(body=share list)`

### BloodHound tools (`tools/bloodhound.py`) — 6 tools

```
bloodhound_start(target)
  Starts Neo4j with data dir at targets/<target>/neo4j/. Idempotent.
  First-time setup: creates data dir, applies BloodHound schema constraints,
  installs APOC plugin if needed, sets initial password (random, stored at
  targets/<target>/neo4j/bolt_password).

bloodhound_stop(target)
  Stops the Neo4j process. Persists data.

bloodhound_status(target?)
  Reports running status, port, data dir, node counts (Users, Computers,
  Groups, OUs, GPOs), last-collection timestamp.
  Without target: lists all targets that have a Neo4j data dir.

bloodhound_collect(target, domain, dc_ip, username, password?, nt_hash?,
                   collection_methods?)
  Wraps `bloodhound-python -c <methods> -d <domain> -u <user> -p <pass>
                          -dc <ip> -ns <ip>`.
  collection_methods default: 'Default,LoggedOn'. 'DCOnly' available
  for stealthier runs.
  Output zip is auto-imported into the per-target Neo4j via the bolt driver.
  Records imported counts to KB notes.

bloodhound_canned(target, query_name, params?)
  query_name enum (15 queries):
    kerberoastable_users, asreproastable_users, shortest_path_to_da,
    computers_where_user_admin, users_with_dcsync, unconstrained_delegation,
    constrained_delegation, password_not_required, computers_no_laps,
    foreign_group_membership, owned_to_high_value, sessions_on_target,
    high_value_targets, domain_admins, kerberos_delegation_summary
  params: dict for placeholder substitution (e.g., {"username": "jdoe@CORP.LOCAL"})

bloodhound_query(target, cypher, params?, allow_writes=False)
  Free-form cypher against the per-target Neo4j. Read-only by default;
  CREATE/MERGE/DELETE/SET/REMOVE require allow_writes=True.
```

**Neo4j lifecycle implementation:** Uses the Nix-installed `neo4j` binary with
`NEO4J_HOME` overridden to point at `targets/<target>/neo4j/`. PID tracked at
`targets/<target>/neo4j/.pid`. Bolt port: 7687 (only one Neo4j may be running
at a time; `bloodhound_start` errors if a different target's Neo4j is running
and prompts the user to stop it first).

### KB read-side tools (`tools/kb.py`) — 7 tools

```
kb_show(target?)
  Single-screen overview: hosts (count, OS breakdown), open ports (top 10),
  valid creds (count + most recent), finding count by severity. If target
  omitted: defaults to the sole target subdirectory under targets/ that
  contains a state.db file, if exactly one exists; otherwise errors with a
  list of available targets.

kb_list_hosts(target)
  All discovered hosts: ip, hostname, OS, domain, is_dc, smb_signing.

kb_list_services(target, host?, port?)
  Services with version/banner. Filterable.

kb_list_creds(target, status?)
  Credentials with status, source, where they work (joined cred_results).
  status filter optional.

kb_add_finding(target, title, severity, description, evidence_paths?, cvss?)
  severity: info | low | medium | high | critical.

kb_add_note(target, body)
  Free-form scratchpad — LLM hypothesis log, leads, observations.

kb_export_report(target, output_path?)
  Generates a markdown report from KB contents in the same style as
  pentest_report_10.13.38.23.md. Default output: targets/<target>/report.md.
```

**Total new tools:** 6 NetExec + 6 BloodHound + 7 KB = **19 tools**.

## Existing tool retrofit

Each retrofitted tool gains a tail block following this template:

```python
# ── KB write (new) ─────────────────────────────────────────────────
try:
    from ..kb import for_target
    from ..kb.parsers import parse_<tool>_output
    kb = for_target(target)
    for fact in parse_<tool>_output(result["stdout"]):
        kb.record_<kind>(fact)
except Exception as e:
    import logging
    logging.getLogger(__name__).warning("KB write failed in <tool>: %s", e)
# ───────────────────────────────────────────────────────────────────
```

**Two non-negotiable rules:**
1. KB writes are best-effort and silent — wrapped in try/except, logged as
   warning, tool returns its normal output regardless. A broken parser must
   never break a working scan.
2. Parsers are pure functions — `parse_X(text: str) -> list[Fact]`. No side
   effects. Independently unit-testable with fixture corpora.

## Profile + authorization model

### `ad` profile

New entry registered in `profiles.py`. Skill set (TUI F1 menu):

```
- "Initial recon"            — nmap top-1000 + smb_enum + ldap_search anon + nbtscan
- "Identify DCs"             — kerberos_enum userenum + ldap_search
- "Spray known wordlist"     — netexec_smb spray (gated by REVERSER_AD_ALLOW_SPRAY)
- "AS-REP roast"             — kerberos_enum asreproast (anon LDAP for userlist)
- "Kerberoast"               — kerberos_enum kerberoast with KB-stored creds
- "Validate creds everywhere" — netexec_*/check_auth across all KB creds
- "Collect BloodHound"       — bloodhound_start → collect → status
- "Find attack paths"        — bloodhound_canned shortest_path_to_da, owned_to_high_value
- "Dump secrets"             — netexec_smb sam/lsa/ntds with valid local-admin
- "Show what we know"        — kb_show + kb_list_creds + kb_list_hosts
- "Generate report"          — kb_export_report
```

### `ad` profile system prompt (key behaviors)

- **Methodology:** assumed-breach internal engagement: enumerate → spray →
  escalate via graph → dump → lateral.
- **Hypothesis-driven loop:** every 5 tool calls, write down (a) current
  hypothesis, (b) cheapest disconfirming experiment, (c) pivot if it fails.
  Do NOT grind the same primitive past 3 failed attempts.
- **KB usage:** before every new attack, call `kb_show` and `kb_list_creds` —
  do not re-derive known facts. Record findings via `kb_add_finding` as soon
  as confirmed, not at the end.
- **Credential lifecycle:** on every newly-validated credential, immediately
  test it across ldap/winrm/mssql/ssh via `check_auth`, then run
  `bloodhound_canned owned_to_high_value` from that user.
- **BloodHound default:** as soon as any valid domain credential exists, run
  `bloodhound_collect`. Then `shortest_path_to_da` is the default next move.
- **Stop conditions:** Domain Admin reached, OR three orthogonal attack paths
  exhausted with no progress (then write a finding + bail).
- **Scope confirmation:** confirm scope (target IPs, domain, time window)
  before the first active tool call.

### `pentest` profile augmentation

Append one paragraph:
> If during recon you discover SMB (445), LDAP (389/636), or Kerberos (88) —
> especially with a Windows / Domain Controller fingerprint — this is an AD
> environment. Stop the generic pentest flow, suggest the user re-run with
> `-p ad`, and prefer the AD-specific tooling for that target.

### Authorization

Every new tool checks `authz.require_pentest_auth()` at function entry, raising
a clear error if neither `.reverser-authorized` nor
`REVERSER_PENTEST_AUTHORIZED=1` is present. Matches the existing pattern.

## Testing

### 1. KB parser unit tests (most critical)

Fixture-driven; every parser gets ≥3 fixture files covering happy path, empty
input, and one edge case (auth failed, malformed output, etc.). Test pattern:

```python
def test_parse_nmap_output_basic():
    text = (FIXTURES / "nmap" / "host_with_smb_and_winrm.txt").read_text()
    facts = parse_nmap_output(text)
    assert len(facts) == 1
    assert facts[0].ip == "10.10.10.5"
    assert any(s.port == 445 and s.service == "microsoft-ds" for s in facts[0].services)
```

**Coverage target:** every parser has at least one happy-path and one
empty-input fixture. 12 parsers × ≥3 fixtures = ~36 fixture files; ~24 test
functions.

### 2. KB integration tests

Use a tmp-path SQLite, exercise the public API:
- `for_target(t)` creates the directory and DB on first call
- `record_host` is idempotent (same host re-recorded does not duplicate)
- Credential status transitions (`untested` → `valid` → `valid on more services`)
- `kb_show` markdown rendering is non-empty given a populated KB
- `kb_export_report` produces valid markdown with all sections

~10 test functions.

### 3. Tool smoke tests (manual, not CI)

`tests/manual/ad_smoke.md` — a 30-minute walkthrough against a known-stable
HTB AD lab box (e.g., Forest, Sauna, Active). Each step lists expected KB
state after the call. Run before declaring this work complete.

### Not testing

- NetExec / bloodhound-python wrapper behavior (trust upstream tools)
- LLM behavior in `ad` profile (no LLM-eval framework in the repo today)
- Neo4j itself

## Rollout order

1. **KB foundation** — `kb/schema.py`, `kb/store.py`, public API + parser
   tests with fixtures pinned first (TDD). Nothing else lands without this.
2. **KB read-side tools** (`tools/kb.py`) — gives a way to inspect during dev.
3. **Parsers + retrofit** — one parser + retrofit at a time, smallest first
   (`nbtscan_scan` → `banner_grab` → `whatweb_scan` → `nmap_scan` → ...).
4. **NetExec tools** — one protocol at a time (`netexec_smb` first as the
   workhorse, then `ldap`, `winrm`, `mssql`, `ssh`, `ftp_wmi`).
5. **BloodHound stack** — Neo4j lifecycle → collector → canned cypher →
   free-form cypher.
6. **Profile + prompts** — `ad` profile registered, `pentest` augmentation.
7. **Manual smoke test** — the HTB walkthrough; fix anything ugly.

After steps 1+2 the system is already useful: existing tools become KB-aware
as they're retrofitted in step 3. Steps 4–5 are the headline AD work.

## Out of scope (explicit anti-list)

To prevent scope creep mid-implementation:

- ❌ SharpHound (.NET collector via execute-assembly) — bloodhound-python is
  sufficient and lower-risk.
- ❌ BloodHound CE web UI / API server / Postgres — only need Neo4j + cypher.
- ❌ Convenience macros (`netexec_spray`, `netexec_dump_secrets`) — per-protocol
  tools cover the surface; layer macros later if needed.
- ❌ Hashcat orchestration — surfaces hashes via `record_artifact`; the
  cracking workflow is a follow-on roadmap item.
- ❌ Automated lateral-movement chaining — the LLM does the chaining via tools;
  no `auto_lateral` macro.
- ❌ Roadmap items #2 (searchsploit/msfvenom), #3 (Playwright wiring), #5
  (hypothesis-loop prompt restructure) — separate work items.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Parser breakage silently empties KB | Logged warnings + manual smoke test; LLM can also notice via `kb_show` returning unexpectedly little |
| Account lockouts during spray | Built-in `REVERSER_SPRAY_MAX` cap (default 3); spray gated behind explicit `REVERSER_AD_ALLOW_SPRAY=1` |
| Multiple Neo4j instances colliding on bolt port 7687 | `bloodhound_start` checks for running instance and errors with clear message; PID tracking via `targets/<target>/neo4j/.pid` |
| Credential leakage to logs | KB stores creds in plaintext (engagement data); log lines redact passwords/hashes; `targets/` directory excluded by `.gitignore` (verify) |
| KB schema migration pain | Lightweight migrations module from day one; schema version pinned in a `meta` table |
| Neo4j memory pressure inside Incus harness | Bump harness profile from 2G → 4G; document local-host minimum 4G |
